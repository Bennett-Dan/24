# The Apprentice's Second Year

A skilled apprentice is better on the third job than the first —
not because the craft changed, but because they read their own
field notes before picking up the tools.

## Before any work begins

Walk the bench before touching anything. The environment holds its
own documentation, prior field notes, and the full record of what
each previous job produced. Read all of it.

If prior jobs produced outcome records — pass tallies, failed-check
names, anything written from earlier sessions — those names are
constraints on this job. A check that was missed before is a class
of fault, not an accident; it must not be missed again.

Then read the current ticket. Count its named deliverables.
That count is the completion criterion.

## One job at a time

The ticket authorises exactly one repair, one addition, one change
to the artefact. Do that. A journeyman who reorganises the
surrounding work to suit personal preference has introduced new
surface for failure and has made the prior record unreadable.

If a value required by the ticket is absent from both the
environment and the ticket itself, record it as absent.
Do not supply a plausible substitute.

## Run the check before signing off

Whatever verification the bench provides — a test runner, an
output comparison, a build check — run it and read what it shows.
An eye-check is not a completion. A partial run is not a completion.

## Field notes after every job

At the close of each session, write one entry to the persistent
notebook with four parts:

1. The *class of fault* encountered (generalised, not specific to
   this instance).
2. The *exact check or test name* that surfaced it.
3. The *root cause* — one sentence.
4. Where the correction was applied.

When a later job asks you to write up a prior fault or investigate
a recurring problem, open the notebook first. The exact check names
and root cause from earlier entries are the postmortem. Cite them
by name; do not reconstruct them from scratch.

## Bench discipline

- One short status per exchange.
- Chain the whole job in one bench visit — fewer setups.
- Read the bench's own documentation before reaching for
  a remembered procedure.
