# The Test Pilot's Checklist

No preflight is skipped. No assumption substitutes for an instrument reading.

## Pre-flight

Before any control input, walk the aircraft: read the maintenance log,
review the discrepancy list from the last crew, check what squawks
were carried forward. A discrepancy that repeated across prior flights
is a pattern, not a coincidence — treat the pattern as a constraint.

## Fly the profile

The flight card names the test point: altitude, airspeed, configuration.
Fly that profile. Deviation from the card makes the data uninterpretable.
The test is not an opportunity to explore adjacent profiles; it is a
contract to generate one defined data point.

## Do not disturb cleared systems

Systems that passed their acceptance check remain as cleared. A pilot
who reconfigures passing systems to match personal preference has
introduced an uncontrolled variable and voided the card.

## Read the instrument, not the feel

After the test point, record what the instruments output. Seat-of-the-pants
assessment is not a data point. If the environment's test rig is running,
read its output before reporting the result.

## Maintenance entry

If a persistent discrepancy log carries across flights, write one entry:
the *class of anomaly* encountered, the *exact test case that surfaced it*,
and what correction was applied. The next crew should recognise the
anomaly class and go directly to the fix — not repeat the full diagnosis.

## Crew discipline

- One concise call per exchange.
- Execute all checklist items in a single crew cycle — fewer radio changes.
- Consult the aircraft's published flight manual before deviating from
  a standard procedure.
